# 协作规范

项目只使用一条稳定主线：`main → 功能分支 → Pull Request → 合并`。不存在 `develop`
分支，也不要把日常改动直接推送到 `main`。

## 开始与分支

1. 开始前执行 `git fetch origin --prune`，确认本地工作区没有会被覆盖的改动。
2. 从最新 `origin/main` 创建短生命周期分支，例如 `feat/unified-input`、`fix/startup` 或
   `codex/docs-update`；分支名应说明目的。
3. 不改写共享历史，不使用 `reset --hard`、`clean` 等方式清理他人的未提交改动；不要提交
   `.env`、密钥、运行数据、数据库、日志或构建产物。
4. 小阶段完成后检查 `git diff`，运行对应测试，使用能说明用户可见结果的中文提交信息，并推送
   当前功能分支。

## 提交前检查

日常小改动至少运行与改动相关的测试；涉及交互、部署、启动或合并前，运行：

```powershell
.\yuwang.ps1 check
```

该入口覆盖后端质量检查、前端 lint/typecheck/unit/build、浏览器验收、生产表面、Compose
配置与 Windows 启动验收。若真实 Provider 测试所需的环境变量或隔离测试密钥未配置，应在
PR 中明确标记“未执行”，不能填写为通过。

Schema、数据库兼容默认值、事件、Tool 或 Provider 契约变更必须说明兼容性，并补充对应的
契约或迁移测试。安全相关变更必须说明授权范围、默认行为、凭据处理、文件/网络影响和回滚
方式；不得把 mock、固定演示答案或任意命令执行带入生产代码或镜像。

## Pull Request

提交 PR 前请同步目标分支的最新状态并处理冲突，但不要直接把功能分支合并进 `main`。使用
仓库模板完整填写改动、真实测试证据、兼容性、安全边界、截图或脱敏日志、真实 Provider
执行情况和已知限制。至少一名维护者审核；架构或安全边界改动还需相应领域维护者审核后再
合并。
