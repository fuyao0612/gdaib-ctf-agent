# 上下文、记忆与完成可信等级

## 上下文构建

Agent 的 `ContextBuilder` 从不可变 TaskSpec、Thread 对话、用户补充、记忆、附件、工具 Schema、当前计划、观察和剩余预算构建请求。安全层与平台层始终置于用户模板之前；对话、附件、网页、补充和工具输出全部标记为不可信数据。

系统读取相关对话窗口，而不是只读取最后一条消息。`context_token_budget`、`observation_char_budget`、AgentProfile 的消息窗口和文本附件上限会真实参与裁剪。发生裁剪时写入 `context_truncated` 事件，记录原因和估算用量；完整原始审计数据不会被覆盖。

`.txt`、`.md`、`.json` 和 `.log` 可作为受限核心上下文读取：只允许受控相对路径，限制读取字节、UTF-8 解码、行数和字符数。二进制附件只进入元数据。附件文本不能修改系统规则，也不是可执行代码或安全工具。

## 线程记忆

记忆按 Thread 持久化，类型包括线程摘要、运行摘要、重要事实和用户补充。AgentProfile 可禁用记忆及限制事实数量；API 支持查看、整体禁用/启用和清除。运行完成会保存简短运行摘要，人工补充会保存为用户输入记忆。记忆进入模型前仍按不可信内容处理。

## 完成模式

- `advisory`：用于总结、解释和规划等纯模型任务。允许无工具结束，但 Run、界面和报告必须显示“模型生成，未经外部验证”，证据等级为 `model`。
- `structured`：模型返回对象后必须通过 AgentProfile 配置的 JSON Schema，证据等级为 `structured`。Schema 不通过时不能完成。
- `evidence`：继续要求候选绑定成功工具调用或其他外部证据，并通过确定性规则，证据等级为 `external`。

Run 分开保存生命周期状态、`validation_status` 和 `evidence_level`。内部“已完成”不等于“外部已验证”。

## 人工补充与恢复

普通模式的 `request_input` 可按配置进入 `waiting_input`，保存完整检查点并释放当前执行。用户通过补充接口提交内容后，系统把它作为不可信上下文和记忆，从 `input_received` 检查点继续。竞赛模式只能按配置自主重规划或明确失败，不会伪装等待。

停止会取消正在进行的模型协程。恢复和重试沿用检查点中的步骤、请求、Token、费用和累计时间，不重置已消耗预算；不确定的非幂等工具调用仍禁止自动重放。
